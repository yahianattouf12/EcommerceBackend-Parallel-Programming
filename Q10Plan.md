# Q10 - Benchmarking & Bottleneck Analysis (خطة + كود جاهز)

> **ملاحظة شفافية مهمة:** لا أستطيع تشغيل السيرفر فعلياً من هذه الجلسة (نفس مشكلة .NET 9 SDK غير المركّب التي واجهناها في Q7) لأقيس أرقام حقيقية بنفسي. كل ما هنا جاهز ليُشغَّل عندك، وسأشرح لك بالضبط كيف تقرأ النتائج اللي رح تطلع عندك، مع توقّع واقعي للفرق بدل أرقام مُلفّقة.

---

## الأسلوب الموصى به

**استخدام الكاش (Redis) الموجود مسبقاً في مشروعك كحل تحسين، مقارنةً مع نفس العملية بدون كاش — مع قياس الفرق عبر سكريبت الضغط `load-test.ps1` الذي بنيناه في Q9.**

**لماذا هذا الخيار بالضبط، وليس شيء جديد؟**
- مشروعك فيه **بالفعل** نظام Caching جاهز بالكامل: `CacheAspect` (AOP عبر Castle DynamicProxy) + `RedisCacheService`، مطبَّق على `ProductService.GetAllProductsAsyncWithCache()` بسمة `[Cache(60)]`.
- وفي نفس الوقت، عندك `ProductService.GetAllAsync()` (بدون أي كاش) — تُستخدم من `GET /api/products` — وتضرب قاعدة بيانات SQL Server في **كل مرة** بدون أي استثناء.
- هذا يعطيك **زوج "قبل/بعد" حقيقي وجاهز بالكامل بدون أي كود جديد للعملية نفسها** — فقط نحتاج نقيسه ونوثّقه، وهذا أسهل بكثير للدفاع عنه أمام الأستاذ من اختراع تحسين مصطنع.
- أضفنا فقط سطرين Logging داخل `CacheAspect.cs` (شرح بالأسفل) لإظهار "Cache HIT" و"Cache MISS" بوضوح في الكونسول وقت العرض — هذا يلبي شرط "إضافة logging/tracing لتحديد الـ bottleneck".
- عندك مسبقاً `PerformanceMiddleware` يسجّل زمن **كل** طلب HTTP تلقائياً (`PERF: {method} {path} executed in {time} ms`) — هذا يغطّي شرط "قياس الزمن" بدون أي تعديل إضافي.
- لقياس الفرق تحت ضغط حقيقي (وليس طلب واحد فقط)، نستخدم نفس `load-test.ps1` من Q9 — فيه دعم جاهز لـ `-Method GET`، فقط نوجّهه مرة لـ endpoint غير المُخزَّن، ومرة للـ endpoint المُخزَّن، بنفس عدد الطلبات ونفس التزامن.

2. **شغّل السيرفر:**
   ```
   dotnet run --urls=http://localhost:5202
   ```
   اترك الكونسول مفتوح — هنا ستظهر سجلات `PERF:` و `[Cache HIT]` / `[Cache MISS]`.

3. **(تم تطبيقه فعلاً)** أضفنا داخل `AOP/CacheAspect.cs` قياس زمن (`Stopwatch`) حول منطق الكاش، مع طباعة:
   - `[Cache MISS] ... executed against the database in X ms` — عند أول استدعاء (أو بعد انتهاء صلاحية الكاش).
   - `[Cache HIT] ... served from Redis in X ms (no DB call)` — لأي استدعاء بعد ذلك خلال 60 ثانية.

4. **حدد الـ Bottleneck:** أرسل عدة طلبات GET متتالية لـ `GET /api/products` (بدون كاش) وراقب سجلات `PERF:` — ستلاحظ أن **كل** طلب يأخذ وقتاً مشابهاً (لأنه يضرب قاعدة البيانات كل مرة)، بعكس `GET /api/products/products-with-cache` الذي يكون سريعاً جداً من الطلب الثاني فصاعداً.

5. **قِس "قبل" (Before) بدون كاش:**
   ```powershell
   .\load-test.ps1 -Url "http://localhost:5202/api/products" -Method GET -TotalRequests 200 -Concurrency 20 -CsvOutput "before-no-cache.csv"
   ```

6. **قِس "بعد" (After) مع الكاش:**
   ```powershell
   .\load-test.ps1 -Url "http://localhost:5202/api/products/products-with-cache" -Method GET -TotalRequests 200 -Concurrency 20 -CsvOutput "after-with-cache.csv"
   ```
   > **مهم:** شغّل هذا الاختبار **مرتين متتاليتين** — الأولى "تُسخّن" الكاش (أول طلب فيها سيكون Cache Miss بطيء نسبياً)، والثانية تُظهر السرعة الحقيقية للكاش وهو "دافئ" بالكامل. للتقرير النهائي، استخدم نتائج **التشغيلة الثانية**.

7. **افتح الملفين `before-no-cache.csv` و `after-with-cache.csv`**، وقارن الأعمدة (`ElapsedMs`, `Success`).

## تحديد الـ Bottleneck (Bottleneck Identification)

**الـ Bottleneck:** كل استدعاء لـ `GET /api/products` (عبر `ProductService.GetAllAsync()`) ينفّذ استعلام `SELECT * FROM Products` على SQL Server **من الصفر** في كل مرة — بدون أي إعادة استخدام لنتيجة سابقة. تحت ضغط متزامن (عدة مستخدمين بنفس اللحظة)، هذا يعني:
- نفس البيانات (التي لم تتغيّر) تُجلَب من القرص/الذاكرة الداخلية لـ SQL Server مرات متكررة.
- كل طلب يستهلك اتصال من Connection Pool الخاص بـ EF Core طوال مدة الاستعلام.
- زمن الاستجابة لا ينخفض أبداً مهما زاد عدد الطلبات المتشابهة، لأنه لا يوجد أي تخزين مؤقت للنتيجة.

---

## التحسين المُطبَّق (Improvement Applied)

**توجيه القراءات المتكررة لنفس البيانات عبر طبقة Redis Cache الموجودة مسبقاً، بمدة صلاحية 60 ثانية، بدل ضرب قاعدة البيانات في كل طلب.**

آلية العمل (`AOP/CacheAspect.cs` + `[Cache(60)]` على `GetAllProductsAsyncWithCache`):
1. أول طلب: لا يوجد مفتاح بالكاش → ينفّذ الاستعلام الحقيقي على قاعدة البيانات (Cache Miss) → يخزّن النتيجة بـ Redis لمدة 60 ثانية.
2. أي طلب لاحق خلال 60 ثانية: يجد القيمة جاهزة بـ Redis (Cache Hit) → يُعيدها **بدون أي اتصال بقاعدة البيانات على الإطلاق**.

**لماذا هذا يحسّن الأداء فعلياً:**
- Redis يخزّن البيانات بالذاكرة (in-memory)، فزمن القراءة منه أقرب لجزء من المايكروثانية، مقارنة باستعلام SQL كامل (parsing + execution plan + I/O + شبكة).
- يُزيل الضغط عن SQL Server بالكامل لهذه القراءات المتكررة، فيُحرَّر Connection Pool لعمليات أخرى أهم (مثل الكتابة/الطلبات).
- النتيجة تتوسّع (scale) بشكل أفضل بكثير مع زيادة عدد المستخدمين المتزامنين، لأن التكلفة لا تكبر مع عدد الطلبات بل تبقى ثابتة تقريباً.

### توقّع واقعي (لتعرف هل أرقامك منطقية، وليست أرقاماً مُقاسة من قبلي)

- استعلام SQL Server بسيط على جدول صغير من جهاز تطوير محلي: عادة **20-100 ميلي ثانية** لكل طلب (يشمل فتح اتصال، تنفيذ، تحويل النتائج).
- قراءة من Redis محلي (Cache Hit): عادة **أقل من 2-5 ميلي ثانية**.
- يعني نسبة تحسّن متوقعة بين **85% إلى 95%+** في متوسط زمن الاستجابة — هذا نطاق منطقي تتوقع أن تقع نتائجك الحقيقية قريباً منه، لكن **الأرقام الفعلية التي تعرضها للأستاذ يجب أن تكون من تشغيلك الحقيقي للسكريبت، لا من هذا التقدير**.

## سيناريو العرض التوضيحي لأستاذك

1. **افتح كونسول السيرفر مباشرة أمام الأستاذ.**
2. **شغّل طلب واحد لـ `GET /api/products`** (من المتصفح أو REST Client) — أشِر إلى سطر `PERF:` بالكونسول، اذكر الزمن.
3. **شغّل طلب واحد لـ `GET /api/products/products-with-cache` مرتين متتاليتين** — أول مرة سترى `[Cache MISS] ... executed against the database` بزمن قريب من السابق، والمرة الثانية سترى `[Cache HIT] ... served from Redis` بزمن أصغر بشكل واضح وملحوظ بالعين قبل حتى النظر للأرقام.
4. **شغّل اختبار الضغط الكامل أمام الأستاذ** (الخطوتين 5 و 6 بالأعلى)، واعرض التقرير النهائي من كل تشغيلة جنباً لجنب.
5. **اعرض الجدول النهائي (Before/After) مع نسبة التحسّن المحسوبة.**

### ما الذي تقوله للأستاذ (شرح مبسّط)

> "لاحظت أن نقطة `GET /api/products` تضرب قاعدة البيانات في كل مرة حتى لو البيانات نفسها لم تتغيّر — وهذا هو الـ bottleneck. استخدمت طبقة الكاش بـ Redis الموجودة مسبقاً بالمشروع لأخزّن نتيجة هذا الاستعلام لمدة 60 ثانية. قِست الفرق بنفس سكريبت الضغط المتزامن من السؤال السابق، على نفس عدد الطلبات ونفس مستوى التزامن، مرة بدون كاش ومرة معه. النتيجة كانت انخفاض واضح في متوسط زمن الاستجابة، لأن أغلب الطلبات أصبحت تُخدَّم من الذاكرة بدل قاعدة البيانات. هذا تحسين عملي وقابل للقياس، وليس نظرياً فقط."
