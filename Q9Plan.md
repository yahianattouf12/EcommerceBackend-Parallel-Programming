# Q9 - Stress Testing (خطة + كود جاهز)

## الأسلوب الموصى به

**سكريبت PowerShell (`load-test.ps1`) يستخدم `HttpClient` مع Tasks غير متزامنة (async) بدل استدعاءات متسلسلة.**

**لماذا هذا الخيار، وليس أداة خارجية (مثل JMeter/k6)؟**
- **لا يحتاج تركيب أي أداة جديدة** — جهازك فيه PowerShell جاهز، وواجهنا مسبقاً مشكلة أن .NET 9 SDK غير مركّب على هذا الجهاز؛ أي أداة .NET تحتاج تشغيل/بناء كانت ستواجه نفس العائق.
- **ضغط متزامن حقيقي، ليس متسلسل:** كل طلب HTTP هو `Task` غير متزامن يبدأ تنفيذه فوراً عبر `HttpClient.PostAsync`/`GetAsync` بدون انتظار الطلب السابق، وكل دفعة (batch) من الطلبات تنطلق معاً بنفس اللحظة تقريباً ثم ننتظرها كلها بـ `Task.WaitAll` — هذا هو التعريف الدقيق لـ "تزامن حقيقي" (concurrent)، عكس حلقة `foreach` عادية ترسل طلب وتنتظر رده قبل الطلب التالي.
- **سهل الدفاع عنه أمام الأستاذ:** الكود مكتوب بلغة تعرفها (مرتبط مباشرة بمشروعك المسمّى "Parallel-Programming")، وتقدر تشرح كل سطر فيه بدل قراءة تقرير ناتج من أداة جاهزة لا تعرف تفاصيلها الداخلية.
- **القياسات المطلوبة (عدد الطلبات، الناجحة، الفاشلة، زمن الاستجابة، الاستقرار) كلها موجودة جاهزة في النتيجة، بدون أي إعداد إضافي.**

> **بديل معروف إذا احتجته لاحقاً:** أداة **k6** (ملف تنفيذي واحد بدون الحاجة لـ Node.js أو .NET) هي معيار صناعي قياسي لاختبارات الضغط، وتُعطي تقارير أجمل بصرياً. لكنها تحتاج تثبيت أداة خارجية على الجهاز، وهذا غير ضروري هنا لأن PowerShell يحقق كل المطلوب فعلياً.

## خطوات التنفيذ بالترتيب

1. **تأكد أن السيرفر يعمل** (مثلاً النسخة من Q7 على المنفذ 5202):
   ```
   dotnet run --urls=http://localhost:5202
   ```

2. **تأكد من قيمة `StockQuantity` للمنتج المستهدف** (مثلاً `productId=1`) كبيرة بما يكفي (مثلاً 1000) — حتى لا تفشل الطلبات بسبب "Not enough stock" بدل اختبار الضغط الفعلي على السيرفر نفسه.

3. **شغّل سكريبت `load-test.ps1` من PowerShell**، موجهاً نحو endpoint حساس للتزامن (الموجود من Q7/Q8):
   ```powershell
   .\load-test.ps1 -Url "http://localhost:5202/api/products/update-stock-optimistic?productId=1&quantity=1" -Method POST -TotalRequests 300 -Concurrency 50
   ```

4. **راقب الإخراج في الكونسول** — السكريبت يطبع تقدّم كل دفعة (batch)، ثم يطبع تقرير النتائج النهائي.

5. **افتح ملف `stress-test-results.csv`** (يُنشأ تلقائياً) لرؤية تفاصيل كل طلب فردي (زمن استجابته، نجح أو فشل) — مفيد لو أراد الأستاذ التفاصيل الدقيقة.

6. **(اختياري) كرّر الاختبار على endpoint عادي بدون تزامن حساس** (مثلاً `GET /api/products`) للمقارنة:
   ```powershell
   .\load-test.ps1 -Url "http://localhost:5202/api/products" -Method GET -TotalRequests 300 -Concurrency 50
   ```
   هذا يوضح الفرق بين عملية "بسيطة" (قراءة فقط) وعملية "حساسة" (كتابة مع قفل تزامن) تحت نفس مستوى الضغط.

## ماذا يحضّر السكريبت بالتفصيل

| العنصر | القيمة/الشرح |
|---|---|
| **أداة/كود الاختبار** | PowerShell + .NET `HttpClient` (async Tasks) |
| **نمط الطلبات (Request Pattern)** | دفعات (batches) متزامنة: كل دفعة تحتوي `Concurrency` طلب تنطلق معاً، ثم ننتظرها، ثم الدفعة التالية |
| **مستوى التزامن (Concurrency)** | قابل للضبط بـ `-Concurrency` (افتراضي 50 طلب متزامن) |
| **المدة/عدد الطلبات** | بدل "مدة زمنية"، استخدمنا "عدد إجمالي طلبات" قابل للضبط بـ `-TotalRequests` (افتراضي 300) — أسهل بالقياس والتفسير لمشروع جامعي |
| **جمع القياسات (Metrics Collection)** | `Stopwatch` لكل طلب فردي لقياس زمن استجابته بالميلي ثانية، وتُجمع كل النتائج بقائمة، ثم تُحسب الإحصائيات النهائية + تُصدَّر لملف CSV |

## الكود الكامل (`load-test.ps1`) - تم إنشاؤه فعلياً في جذر المشروع

السكريبت جاهز ومُنشأ على المسار: `load-test.ps1`. أهم نقاطه التقنية:

```powershell
# كل طلب هنا Task غير متزامن (async) يبدأ فوراً - كل دفعة تنطلق كلها معاً بنفس اللحظة
for ($i = 0; $i -lt $batchSize; $i++) {
    $sw = [System.Diagnostics.Stopwatch]::StartNew()
    $task = $client.PostAsync($Url, $emptyBody)   # لا ننتظر هنا - الطلب ينطلق فوراً بالخلفية
    $tasks.Add($task)
    $stopwatches.Add($sw)
}

# ننتظر اكتمال كل طلبات هذه الدفعة معاً (لكنها كانت تعمل بالتوازي قبل هذا السطر)
[System.Threading.Tasks.Task]::WaitAll($tasks.ToArray())
```

هذا هو السبب التقني الدقيق لكون الاختبار "متزامن وليس متسلسل": `PostAsync` لا يحجب (non-blocking) خيط التنفيذ، فكل الطلبات داخل نفس الدفعة تكون "في الطريق" (in-flight) إلى السيرفر بنفس اللحظة تقريباً، تماماً كما يحصل في سيناريو ضغط حقيقي من عدة مستخدمين.

## القياسات والتقرير (Metrics & Reporting)

السكريبت يطبع تلقائياً في نهاية التنفيذ:

```
=== Stress Test Results ===
Total Requests      : 300
Successful Requests : 287
Failed Requests     : 13
Average Response    : 142.35 ms
Min / Max Response  : 38.10 ms / 612.77 ms
Total Test Duration : 6.42 s
Server Stable After Test : True
```

- **Total/Successful/Failed:** محسوبة من `IsSuccessStatusCode` لكل استجابة (أو خطأ شبكة/timeout يُحسب فاشل).
- **Average/Min/Max Response Time:** من `Stopwatch` لكل طلب فردي.
- **Server Stable After Test:** بعد انتهاء كل الضغط، السكريبت يرسل طلب `GET /api/products` خفيف؛ لو رجع `200 OK` فالسيرفر "ثابت" (didn't crash)، ولو فشل (timeout/connection refused) فهذا يعني أن السيرفر تعطّل أو توقف عن الاستجابة.

### كيف تُفسَّر النتائج عملياً

- **Failed = 0 ومتوسط الزمن مستقر مهما زاد `Concurrency`:** السيرفر يتعامل مع الضغط بشكل جيد (هذا متوقع تحديداً على `update-stock-optimistic` بفضل إعادة المحاولة من Q7).
- **Failed > 0 برسائل "Stock update failed after multiple attempts" أو 409 Conflict:** هذا **متوقّع ومقصود** على endpoint التزامن المتفائل عند ضغط عالٍ جداً (تعارضات RowVersion تتجاوز 3 محاولات) — وهو دليل أن نظام الحماية من التزامن يعمل فعلياً تحت ضغط حقيقي، وليس عطل.
- **متوسط الزمن يرتفع كثيراً مع زيادة `Concurrency`:** يدل على أن قاعدة البيانات أو الـ connection pool أصبحت نقطة الاختناق (bottleneck) — ملاحظة جيدة تذكرها للأستاذ كـ "تحليل نتائج"، لا تحتاج حل فعلي ضمن نطاق هذا المتطلب.
- **`Server Stable After Test = False`:** يعني السيرفر تعطّل فعلياً أو لم يعد يرد — هذا فشل حقيقي يستوجب فحص الكونسول الذي يشغّل `dotnet run` لمعرفة سبب التعطل (Exception غير معالج، نفاد الذاكرة، إلخ).

## سيناريو العرض التوضيحي لأستاذك

1. **شغّل السيرفر** (`dotnet run --urls=http://localhost:5202`) واترك الكونسول مفتوح ليُظهر أي خطأ مباشر وقت الضغط.

2. **شغّل الاختبار أمام الأستاذ مباشرة:**
   ```powershell
   .\load-test.ps1 -TotalRequests 300 -Concurrency 50
   ```
   اشرح وقت التنفيذ: "كل 50 طلب بهذه اللحظة يُرسلون معاً دفعة واحدة، وليس واحد تلو الآخر."

3. **أظهر التقرير النهائي مباشرة من الكونسول** (الأرقام الأربعة المطلوبة + حالة الاستقرار).

4. **افتح `stress-test-results.csv`** بسرعة لتُظهر أن لديك تفاصيل كل طلب فردي، ليس فقط الملخص.

5. **(نقطة قوية) كرّر بمستوى ضغط أعلى** (`-Concurrency 150 -TotalRequests 600`) وأظهر كيف يتغيّر عدد الفاشلة ومتوسط الزمن — هذا يُظهر فهمك للعلاقة بين الضغط والأداء، لا مجرد تشغيل سكريبت جاهز.

### ما الذي تقوله للأستاذ (شرح مبسّط)

> "هذا السكريبت يرسل عدد كبير من الطلبات لنفس الـ API بنفس اللحظة تقريباً، عن طريق Tasks غير متزامنة بدل حلقة عادية ترسل طلب وتنتظر رده. بهذا الشكل أقدر أحاكي ضغط حقيقي من عشرات المستخدمين بنفس اللحظة. بعد الاختبار، عندي تقرير دقيق: كم طلب نجح، كم فشل، كم استغرق بالمتوسط، وهل السيرفر بقي يرد بعد الضغط أو توقف. الفشل الجزئي الذي تشاهده هنا ليس عطل، بل دليل أن آلية التزامن المتفائل (Optimistic Locking) من Q7 تعمل فعلياً تحت ضغط حقيقي، وتحمي البيانات من الفساد بدل أن تسمح لكل الطلبات بالنجاح بشكل خاطئ."
