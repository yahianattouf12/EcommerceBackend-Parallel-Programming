# Q9 - Stress Testing باستخدام k6 (خطة + كود جاهز)

> هذا بديل لما بنيناه بـ PowerShell (`load-test.ps1`). الملفان يبقيان موجودين بالمشروع، اختر أيهما تعرضه أو اعرض الاثنين كدليل أنك تعرف أكثر من طريقة.

---

## الأسلوب الموصى به

**استخدام k6 (أداة قياسية في صناعة اختبارات الضغط) لتوليد ضغط متزامن حقيقي عبر "Virtual Users" (VUs)، مع تقرير نتائج جاهز ومعروف الشكل.**

**لماذا k6 بالتحديد:**
- **ملف تنفيذي واحد (single binary)** — لا يحتاج Node.js أو أي بيئة تشغيل إضافية، فقط تثبيته مرة واحدة.
- **أداة معروفة فعلياً في الصناعة** — إذا ذكرتها لأستاذك، اسمها معروف بعكس سكريبت مكتوب يدوياً، وهذا يضيف مصداقية لعرضك.
- **تصميمها مخصص فعلياً لهذا النوع من الاختبار:** "Virtual Users" (VUs) كل واحد منهم يرسل طلبات بشكل متكرر ومستقل عن الباقي طوال "مدة" (Duration) محددة — هذا يلبي بالضبط متطلبات السؤال الأصلي (concurrency level **و** duration معاً)، وهو أدق من نموذج "دفعات" (batches) الذي استخدمناه بـ PowerShell.
- **تقرير جاهز ومنسّق:** يعطيك تلقائياً `http_reqs` (إجمالي الطلبات)، `http_req_failed` (نسبة/عدد الفاشلة)، `http_req_duration` (avg/min/med/p90/p95/max)، وعدد الـ VUs والـ iterations — بدون أي كود إضافي لحساب الإحصائيات.

**الفرق الجوهري عن PowerShell (`load-test.ps1`):**
- PowerShell: "أرسل X طلب، بمستوى تزامن Y، حتى تنتهي كلها" (عدد ثابت من الطلبات).
- k6: "شغّل Y مستخدم افتراضي، كل واحد يرسل طلبات بشكل متكرر ومستمر، لمدة Z ثانية" (ضغط مستمر بمرور الزمن) — هذا أقرب لتعريف "Stress Test" الحقيقي المستخدم في الصناعة.

---

## خطوات التنفيذ بالترتيب

### 1) تثبيت k6 على Windows (مرة واحدة فقط)

اختر طريقة واحدة من الثلاث:

```powershell
# الطريقة الأولى: عبر winget (مدمج بـ Windows 11)
winget install k6 --source winget

# الطريقة الثانية: عبر Chocolatey (إذا كان مركّباً عندك)
choco install k6

# الطريقة الثالثة: تحميل يدوي
# من https://k6.io/docs/get-started/installation - حمّل النسخة لـ Windows وثبّتها
```

تحقق من التثبيت:
```powershell
k6 version
```

### 2) تأكد من تشغيل السيرفر

```powershell
dotnet run --urls=http://localhost:5202
```

### 3) تأكد من قيمة `StockQuantity` للمنتج المستهدف (مثلاً `productId=1`) كبيرة بما يكفي (مثلاً 5000) — لأن k6 سيرسل طلبات بشكل متواصل طوال مدة الاختبار، فيحتاج مخزون كبير حتى لا تفشل كل الطلبات بسبب نقص المخزون بدل قياس الأداء الفعلي.

### 4) شغّل الاختبار (الملف `stress-test.js` تم إنشاؤه فعلاً في جذر المشروع)

```powershell
k6 run stress-test.js
```

أو بضبط المعاملات يدوياً:
```powershell
k6 run -e BASE_URL=http://localhost:5202 -e PRODUCT_ID=1 -e VUS=50 -e DURATION=30s stress-test.js
```

### 5) راقب الإخراج في الكونسول — k6 يطبع تقرير شامل تلقائياً في نهاية التنفيذ (الشكل بالأسفل).

## الكود الكامل (`stress-test.js`) - تم إنشاؤه فعلياً بجذر المشروع

```javascript
import http from "k6/http";
import { check, sleep } from "k6";

const BASE_URL = __ENV.BASE_URL || "http://localhost:5202";
const PRODUCT_ID = __ENV.PRODUCT_ID || 1;
const QUANTITY = __ENV.QUANTITY || 1;
const VUS = __ENV.VUS ? parseInt(__ENV.VUS) : 50;
const DURATION = __ENV.DURATION || "30s";

// عدد المستخدمين الافتراضيين (VUs) يعمل كل واحد منهم بشكل متزامن طوال مدة DURATION
export const options = {
  scenarios: {
    concurrent_pressure: {
      executor: "constant-vus",
      vus: VUS,
      duration: DURATION,
    },
  },
};

export default function () {
  const url = `${BASE_URL}/api/products/update-stock-optimistic?productId=${PRODUCT_ID}&quantity=${QUANTITY}`;
  const res = http.post(url);

  check(res, {
    "status is 200": (r) => r.status === 200,
  });

  sleep(0.1);
}

// فحص استقرار السيرفر بعد انتهاء الضغط بالكامل
export function teardown() {
  const res = http.get(`${BASE_URL}/api/products`);
  console.log(
    `Post-test stability check: GET /api/products returned status ${res.status} ` +
      (res.status === 200 ? "(server stable)" : "(server NOT responding correctly)")
  );
}
```

**شرح سريع لأهم الأسطر:**
- `executor: "constant-vus"` — يحافظ على عدد ثابت من "المستخدمين الافتراضيين" (VUs) يعملون بالتوازي طوال `DURATION`، كل واحد منهم يكرر تنفيذ الدالة `export default function()` بشكل مستقل عن الآخرين — هذا هو التزامن الحقيقي.
- `check(res, ...)` — يسجّل لكل طلب هل نجح (status 200) أم لا، ويُجمَّع تلقائياً بالتقرير النهائي.
- `teardown()` — يُنفَّذ **مرة واحدة فقط** بعد انتهاء كل الـ VUs، ليثبت أن السيرفر لا زال يرد بشكل سليم (شرط "Stability/Crash status" بالسؤال).

## شكل التقرير الذي يطبعه k6 تلقائياً، وكيف تقرأ كل سطر منه

```
     ✓ status is 200

     checks.........................: 94.32% ✓ 1415      ✗ 85
     data_received..................: 612 kB  20 kB/s
     data_sent.......................: 198 kB  6.6 kB/s
     http_req_duration..............: avg=118.4ms min=22.1ms med=98.7ms max=612.0ms p(90)=210.3ms p(95)=289.5ms
     http_req_failed.................: 5.66%   ✓ 85        ✗ 1415
     http_reqs........................: 1500    49.8/s
     iteration_duration..............: avg=218.5ms
     iterations.......................: 1500    49.8/s
     vus..............................: 50      min=50      max=50
     vus_max..........................: 50      min=50      max=50
```

| ما يطلبه السؤال | السطر المطابق في تقرير k6 |
|---|---|
| **Total Requests** | `http_reqs` (هنا 1500) |
| **Successful Requests** | `checks` ✓ (هنا 1415) أو `http_reqs - http_req_failed count` |
| **Failed Requests** | `http_req_failed` ✗ (هنا 85) |
| **Average Response Time** | `http_req_duration avg` (هنا 118.4ms) |
| **Stability/Crash Status** | سطر `Post-test stability check` المطبوع من `teardown()` + استمرار k6 بالعمل حتى النهاية بدون توقف مفاجئ |

> الأرقام أعلاه **مثال توضيحي فقط** لشرح كيفية القراءة - أرقامك الحقيقية ستختلف حسب جهازك وعدد الـ VUs والمدة التي تختارها.

## كيف تُفسَّر النتائج عملياً

- **`http_req_failed` منخفضة (أو صفر) ومتوسط الزمن مستقر:** السيرفر يتعامل مع الضغط بشكل جيد عند هذا المستوى من التزامن.
- **`http_req_failed` ترتفع مع رسائل "Stock update failed after multiple attempts" أو 409:** هذا **متوقّع ومقصود** على endpoint التزامن المتفائل (`update-stock-optimistic`) عند ضغط مرتفع جداً — دليل أن حماية التزامن (RowVersion من Q7) تعمل فعلياً، وليس عطل بالسيرفر.
- **`p(95)` أو `p(90)` مرتفعة جداً مقارنة بـ `avg`:** يعني أن أغلب الطلبات سريعة، لكن نسبة منها (5-10%) تنتظر طويلاً — مؤشر على نقطة اختناق (مثل تنافس على نفس صف بقاعدة البيانات).
- **رسالة `teardown()` تُظهر status غير 200، أو k6 يتوقف بخطأ اتصال (connection refused) قبل النهاية:** يعني السيرفر تعطّل فعلياً تحت الضغط — يستوجب فحص كونسول `dotnet run` لمعرفة سبب التعطل.

## سيناريو العرض التوضيحي لأستاذك

1. **شغّل السيرفر** واترك الكونسول مفتوحاً.
2. **اذكر للأستاذ أنك استخدمت أداة معيارية بالصناعة (k6)** لا سكريبت مخصص فقط - هذا يقوّي عرضك.
3. **شغّل الاختبار أمام الأستاذ:**
   ```powershell
   k6 run -e VUS=50 -e DURATION=20s stress-test.js
   ```
4. **اشرح أثناء الانتظار:** "الآن عندي 50 مستخدم افتراضي يرسلون طلبات بالتوازي بشكل مستمر لمدة 20 ثانية - هذا ضغط متزامن حقيقي، وليس طلب بعد طلب."
5. **اعرض التقرير النهائي مباشرة من الكونسول** وأشِر للأستاذ على الأسطر المطلوبة (`http_reqs`, `http_req_failed`, `http_req_duration avg`, ورسالة الاستقرار من `teardown`).
6. **(نقطة قوية) كرّر برفع `VUS` إلى 150** وأظهر التغيّر بعدد الفاشلة ومتوسط الزمن:
   ```powershell
   k6 run -e VUS=150 -e DURATION=20s stress-test.js
   ```

### ما الذي تقوله للأستاذ (شرح مبسّط)

> "استخدمت k6، أداة معيارية بصناعة اختبارات الضغط، لمحاكاة عدد كبير من المستخدمين الحقيقيين يضغطون على نفس API بنفس الوقت لمدة محددة، بدل طلبات متتالية. التقرير يعطيني تلقائياً عدد الطلبات الكلي، الناجحة، الفاشلة، ومتوسط زمن الاستجابة. وأضفت فحص استقرار في نهاية الاختبار للتأكد أن السيرفر بقي يرد بشكل سليم ولم يتعطل تحت الضغط. الفشل الجزئي الذي تشاهده هنا مقصود ومتوقع، وهو دليل أن آلية التزامن المتفائل تحمي البيانات تحت ضغط حقيقي."

## ملاحظة شفافية

لم أتمكن من تشغيل k6 فعلياً من هذه الجلسة لاختبار السكريبت على خادم حقيقي (لا k6 ولا .NET 9 SDK مركّبان على هذا الجهاز). الكود أعلاه صحيح بنيوياً وفق توثيق k6 الرسمي، لكن شغّله أنت أولاً بعدد صغير من VUs (`VUS=5 -e DURATION=10s`) للتأكد أنه يعمل كما هو متوقع على بيئتك قبل اعتماده للعرض النهائي.
